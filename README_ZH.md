## 什麼是 Skill？

**Skill（技能）** 是一個可重複使用的 AI Agent 工作流單元，通常包含以下元素：

- **明確的目標**：定義成功的樣貌以及如何衡量
- **輸入/輸出架構**：明確定義的數據結構與格式
- **步驟與限制條件**：明確的指令與邊界
- **工具使用慣例**：如何以及何時使用特定工具
- **測試與評估標準**：驗證正確性與品質的方法

Skill 讓 AI Agent 能夠在不同平台和情境中，一致且可靠地執行專業任務。

## 依平台分類

### Codex

**Codex** 是 OpenAI 的 AI 驅動程式設計助手，支援 CLI 和 IDE 擴展。Codex Skills 透過打包指令、資源和可選腳本來擴展平台的任務專用能力。

Skills 以資料夾形式封裝可重複使用的工作流程；通用結構請參考「[創建技能](#創建技能)」的「[技能結構](#技能結構)」。

**安裝位置：**

Skills 按階層載入（優先順序由高到低）：

| 範圍   | 位置                       | 使用場景               |
| ------ | -------------------------- | ---------------------- |
| REPO   | `$CWD/.codex/skills`       | 當前目錄的任務專用技能 |
| REPO   | `$CWD/../.codex/skills`    | 巢狀倉庫的共享技能     |
| REPO   | `$REPO_ROOT/.codex/skills` | 組織範圍的技能         |
| USER   | `$CODEX_HOME/skills`       | 跨專案的個人技能       |
| ADMIN  | `/etc/codex/skills`        | 所有使用者的系統級技能 |
| SYSTEM | 內建                       | 預設技能               |

**創建技能：**

*使用 Skill Creator：*
```bash
$skill-creator
```
內建工具會引導您定義觸發條件和結構。

*手動創建：* 依照「[創建技能](#創建技能)」的「[基礎技能模板](#基礎技能模板)」建立；Codex 對前置資料的常見限制是 `name` 最多 100 字元、`description` 最多 500 字元，完成後重新啟動 Codex 以載入。

**調用技能：**
- **顯式調用**：輸入 `$skill-name` 或使用 `/skills` 命令
- **隱式調用**：Codex 根據描述匹配自動啟動

**安裝額外技能：**
```bash
$skill-installer linear
$skill-installer notion-spec-to-implementation
```

相關文件請參考「[資源](#資源)」的「[官方文件](#官方文件)」。

### Claude Code

**Claude Code** 是 Anthropic 官方的 CLI 工具，將 Claude 的能力直接帶入您的終端機和開發工作流程。Agent Skills 透過組織化的指令、指令碼和資源來擴展 Claude 的任務專用能力。

**什麼是 Agent Skills：**

Agent Skills 是模組化功能，可擴展 Claude 的能力。每個 Skill 封裝了指令、中繼資料和可選資源，Claude 在相關時會自動使用。Skills 遵循開放 Agent 技能標準，並可在 Claude.ai、Claude Code、Claude API 和 Claude Agent SDK 中使用。

**為什麼使用 Skills：**
- **專門化 Claude**：為特定領域任務量身定制功能
- **減少重複**：建立一次，自動使用
- **組合功能**：結合 Skills 建立複雜工作流程

通用 Skill 結構請參考「[創建技能](#創建技能)」的「[技能結構](#技能結構)」。

**YAML 前置資料（必需）：**
```yaml
---
name: your-skill-name
description: 此 Skill 的功能及使用時機
---
```

要求：
- `name`：最多 64 個字元，僅小寫字母/數字/連字號，無 XML 標籤，無保留字
- `description`：最多 1024 個字元，非空，無 XML 標籤

**漸進式揭露 - 三個載入級別：**

Skills 使用三級載入系統來有效管理上下文：

| 級別                 | 何時載入         | 代幣成本                 | 內容                                           |
| -------------------- | ---------------- | ------------------------ | ---------------------------------------------- |
| **級別 1：中繼資料** | 始終（在啟動時） | 每個 Skill 約 100 個代幣 | YAML 前置資料中的 `name` 和 `description`      |
| **級別 2：指令**     | 觸發 Skill 時    | 少於 5k 個代幣           | 包含指令和指導的 SKILL.md 主體                 |
| **級別 3+：資源**    | 根據需要         | 實際上無限制             | 透過 bash 存取的捆綁檔案，在讀取前無上下文成本 |

Claude 只載入每個任務所需的內容，保持上下文使用高效。

**安裝位置：**

*用於 Claude Code（使用者級別）：*
```bash
# 創建使用者級別的技能目錄
mkdir -p ~/.config/claude-code/skills/

# 將技能資料夾複製到技能目錄
cp -r your_skill_name ~/.config/claude-code/skills/

# 驗證技能是否被識別
claude --list-skills
```

*用於專案專用：*
```bash
# 創建專案級別的技能目錄
mkdir -p .claude/skills/

# 將技能複製到專案
cp -r your_skill_name .claude/skills/
```

`.claude/skills/` 中的 Skills 僅適用於該特定專案，而 `~/.config/claude-code/skills/` 中的 Skills 全域可用。

**創建 Skills：**

依照「[創建技能](#創建技能)」的「[基礎技能模板](#基礎技能模板)」建立 SKILL，並注意上方的 `name` / `description` 限制；完成後複製到對應目錄並用 `claude --list-skills` 驗證。

**Skills 如何被發現：**

1. **啟動**：載入所有 Skill 的中繼資料（名稱 + 描述）
2. **任務匹配**：請求匹配描述時自動觸發
3. **按需載入**：需要時才讀取 SKILL.md 與其他資源檔

**最佳實踐：**
- 保持 SKILL.md 主體在 500 行以下
- 對大型 Skills 使用漸進式揭露
- 撰寫清楚說明「做什麼」和「何時使用」的描述
- 如果使用多個模型，請使用 Haiku、Sonnet 和 Opus 測試
- 僅使用來自受信任來源的 Skills（安全性）

4. **包含範例**：真實世界的使用示範
5. **徹底測試**：在不同場景中驗證
6. **保持簡單**：避免不必要的複雜性
7. **版本控制**：追蹤變更和更新
8. **安全優先**：絕不暴露敏感數據

## 設計模式

本節提供的是「撰寫 Skill / 工作流程時可套用的流程模板」，用來提高可驗收性、可重現性與安全性；每個 Skill 不需要全用，依任務挑選 1–2 個模式作為骨架即可。

### 規劃 → 執行 → 驗證
將任務分為三個階段：規劃方法、執行步驟、驗證結果。

何時使用：任務容易跑偏、需要明確驗收標準時。

### 規格優先工作流程
在執行之前定義輸入、輸出和限制。這確保了清晰性並防止範圍蔓延。

何時使用：輸入/輸出格式很重要，或需求容易變動時。

### 工具優先工作流程
在可能的情況下優先使用結構化工具而非自由格式文本。工具提供更好的錯誤處理和驗證。

何時使用：有可用的 CLI/API/測試/格式化工具可直接驗證結果時。

### 批判者/審查者迴圈
針對評分標準或檢查清單實施自我檢查。讓技能審查自己的輸出品質。

何時使用：輸出需要嚴格符合格式、規範或檢查清單時。

### 最小變更
對於程式碼編輯，進行必要的最小安全變更。避免超出直接需求的重構。

何時使用：改程式碼、設定或自動化腳本，且希望降低副作用風險時。

### 輕量 RAG
進行研究時，引用並註明來源。保持從結論到證據的可追溯性。

何時使用：需要查證資訊、引用文件內容或降低「憑空編造」風險時。

### 可組合性
設計技能以協同工作。一個技能的輸出應該可以作為另一個技能的輸入使用。

何時使用：你打算把多個 Skill 串成一條工作流時。

### 冪等性
技能在使用相同輸入多次執行時應產生相同的結果。

何時使用：希望同一輸入可重跑、可回歸測試、可預期產出時。

## 評估與品質

### 品質檢查清單

每個技能都應解決以下維度：

- **清晰的 I/O 契約**：明確定義的輸入和輸出以及範例
- **邊界情況**：處理邊界條件和錯誤
- **確定性**：盡可能的可預測行為
- **停止條件**：完成或失敗的明確標準
- **安全注意事項**：意識到機密、注入風險和數據安全
- **範例執行**：至少一個完整的範例和預期輸出
- **文件**：為使用者和維護者提供清晰的指令

### 測試建議

- **單元測試**：針對個別技能元件
- **整合測試**：針對技能與工具的互動
- **Golden Files**：用於回歸測試的參考輸出
- **使用者驗收**：與目標使用者的真實世界驗證

### 品質指標

- **成功率**：成功執行的百分比
- **錯誤率**：失敗或異常的頻率
- **效能**：執行時間和資源使用
- **使用者滿意度**：來自技能使用者的回饋

## 安全與風險控管

### 最佳實踐

- **絕不貼上機密**：使用環境變數和密鑰管理器
- **將外部內容視為不可信**：驗證並清理所有輸入
- **最小權限工具存取**：授予最低必要權限
- **記錄並審查操作**：為自動化操作維護稽核軌跡
- **驗證輸出**：在提交或發布之前檢查結果
- **速率限制**：為依賴 API 的技能實施節流
- **數據隱私**：尊重使用者數據和隱私法規

### 常見風險

- **提示注入**：劫持技能行為的惡意輸入
- **數據洩漏**：意外暴露敏感資訊
- **意外操作**：技能執行超出其範圍的操作
- **資源耗盡**：技能消耗過多運算或記憶體
